## 1. 這是什麼工具、給誰用

這是一個「組合屋屋頂叫料計算機」：使用者是組合屋／鋼構屋／鐵工的**工班師傅，不是工程師**。他在畫布上排好幾間組合屋（3×6m、4×6m 這類固定規格模矩），設定單斜／雙斜、坡度（分水比）、延伸出簷，按一個鈕就得到可以直接複製貼到 LINE 給料商的叫料單（C型鋼支數＋長度、浪板片數、包角、中脊配件）。所有輸出必須讓非工程師直接看懂、直接能用——這也是為什麼這個專案花了大量力氣在「畫面上的標籤會不會誤導人」而不只是「公式對不對」。歷史上真正造成損失的 bug，一半以上是**標籤／視覺誤導**問題，不是算術問題。

上線位置：https://lengkaisue.github.io/ping-calc/combo-roof-v2.html（GitHub repo: lengkaisue/ping-calc；注意 GitHub Pages 建置實測約 9 分鐘，不是 1 分鐘）。

## 2. 資料模型（runtime 實際物件）

### 2.1 `buildings[]` — 畫布上每一棟建物

宣告在 script 開頭（`let buildings=[]`），元素由 `addBuildings()` 建立：

```js
{x, y, w, d, type, state, groupId, dir}
```

| 欄位 | 意義 | 千萬要注意的事 |
|---|---|---|
| `x, y` | 世界座標（cm），左上角 | 拖曳時吸附 10cm 格（`onMove`）並邊對邊吸附（`snapEdges`） |
| `w, d` | **原始規格尺寸**（cm），例如選 4×6 米就是 `w=400, d=600` | **永遠不變**，轉向也不變。畫面上任何「原始尺寸」顯示都應該取這裡 |
| `type` | `'modular'`（組合屋）或 `'deck'`（塑木平台） | deck 不參與屋頂算料、沒有 `dir` 概念，只算面積 |
| `state` | `1`＝正向、`2`＝轉向 90° | 只影響**畫布上的外觀佔位**。轉向後畫面寬深由 `bDims()` 交換 |
| `groupId` | 手動分組 id，未分組為 `null` | 有 `groupId` 的建物不顯示個別方向箭頭，方向由 `manualGroups` 的中脊線決定 |
| `dir` | **每棟自己的排水方向**：`'col'`＝前後斜（沿畫面 D／直向）、`'row'`＝左右斜（沿畫面 W／橫向） | **這是 per-building 屬性，不是全域設定**。UI 上的 `#roofDir` 下拉選單只決定「新增建物的預設值」。切換入口是畫布上每棟右上角的 ↕/↔ 小圖示（`_dirBtnRects` 命中判斷，在 `onDown` 裡） |

**`bDims(b)`（約 :330）是全檔最重要的一行：**

```js
function bDims(b){ return b.state===2?{bw:b.d,bd:b.w}:{bw:b.w,bd:b.d}; }
```

`bw/bd` 是「轉向後、畫面上的實際寬／深」。**所有幾何計算（碰撞、貼齊、分組、外框）都用 `bw/bd`；所有「給人看的原始尺寸標籤」都用 `w/d`。** 把這兩者搞混，就是本專案反覆踩的第一大坑（見第 4 節）。

### 2.2 `manualGroups[]` — 手動分組（強制合併）

```js
{id, ridge:{x1,y1,x2,y2}, ridgeColMarks:[t, ...]}
```

- 由「Shift+點擊（或手機多選模式）選 ≥2 棟 → 強制合併為一組」建立（`onGroupActionClick`）。初始中脊線自動取 bounding box 最長邊方向過中點，兩端有可拖曳把手（`hitRidgeEndpoint`）。
- **中脊線方向決定整組的排水方向**（`dx>=dy → 'col'`，在 `groupBuildings()` 裡判斷），該組成員的個別 `dir` 與 ↕/↔ 箭頭全部失效／不顯示。
- `ridgeColMarks`：使用者在中脊線上點出的「強制落柱點」，每個值是**沿線 0~1 的比例位置**（不是絕對 cm）——這樣拖動端點調整中脊線時標記會等比例跟著走。有標記時中脊柱組數＝標記數，蓋掉自動公式（設計理念見 memory `feedback_manual_override_over_formula`：結構例外多的細節，讓 domain expert 手動指定，不要硬做全自動公式）。
- 手機端點附近容易誤觸拖曳把手，所以有 `addEndMarks()` 按鈕直接在 t=0.03/0.97 加標記（內縮避開把手判定半徑）。
- 分組成功後會**自動關閉多選模式**（`onGroupActionClick` 尾端）——不關的話後續點中脊線會被多選模式攔截吃掉，曾造成用戶回報「中柱落點功能壞了」。
- 成員不足 2 間時 `cleanupGroups()` 整組解散。

### 2.3 `groupBuildings(mods)` 的輸出——兩個消費端共用的唯一契約

`groupBuildings()`（約 :938）把 `buildings` 裡的組合屋整併成 roofUnits：

```js
{w, d, count, isRect, members, manual, dir?, ridgeColMarks?}
```

- `w, d`：這個屋頂單元的外框尺寸（cm，已含轉向效果，因為內部用 `bDims()`）。
- `count`：成員數；`members`：原始 building 物件的參照陣列。
- `manual: true` 的 unit 才有 `dir`（來自中脊線方向）與 `ridgeColMarks`。
- 非手動 unit 的方向由消費端自己取 `u.members[0].dir`（自動合併群組的箭頭點擊會同步寫入全體成員的 `dir`，所以取第一棟即可代表）。

分組規則三層：① 有 `groupId` 的直接用 bounding box 當一個 unit（不做矩形檢查）；② 其餘用 Union-Find 判斷「邊對邊完全貼齊且同寬/同深」，貼齊集合的面積總和＝外框面積才視為合併矩形；③ 貼在一起但拼不成矩形（L 型缺角）→ 拆回逐間各算，**不會**用錯誤外框硬套。十字/L 型想合併算，就是要走手動分組。

**這個輸出被三個地方消費，且必須永遠一致：** `calcMats()`（叫料）、`draw3D()`→`calc3DUnitDims()`（3D 示意）、`draw()`（2D 俯視圖的合併預覽虛線框＋合併群組的共用方向箭頭）。

## 3. 核心公式清單：公式、所在位置、驗證狀態

**驗證狀態分三級，這個分級決定你改它之前需要多大的謹慎：**

- ✅ **實單驗證**：拿真實案場叫料單（4×6m 單斜案場，用戶提供 mm 單位實單）具體數字反推核對過，輸入輸出完全吻合。**改這些公式前必須先有新的具體數字證據，不接受「看起來更合理」。**
- 👁 **視覺驗證後修正**：從未被實單驗證，長期是「聽起來合理的假設」，直到 3D 圖把它畫出來、用戶一眼看出不對才修。目前值來自用戶口頭的現場常規，**沒有實單背書，遇到不同案場尺寸要重新確認**。
- ❓ **未驗證**：純推論或沿用舊版，從未被任何真實數字核對過。改動風險最低、但正確性也最沒把握。

### 3.1 單斜（rType===1）——2026-07-07 用 4×6m 真實案場整單對平，是全工具驗證最完整的部分

實單基準案例（記住這組數字，回歸測試就用它）：**400×600cm、轉向 state=2（畫面 600 寬 ×400 深）、dir='col' 前後斜、延伸前 50/後 30/左 30/右 30、坡度 1 分水、C型鋼 100 型 2.3mm、浪板 75cm 寬**。實單：斜樑 465cm×4、橫梁 660cm×8（間距 77.5）、浪板 480cm×9 片、左右包角 490×2、橫向包角 340×2、中柱 40cm×4。

| 項目 | 公式（位置：`calcMats()` 內 rType===1 分支，約 :1066-1118） | 驗證狀態 |
|---|---|---|
| 板材長 | `sp = round(dir==='col' ? extD : extW)`，ext 含延伸 | ✅ 400+50+30=480 |
| 斜樑長 | `railCm = sp - 15`（滴水處板子多出樑 15cm） | ✅ 480−15=465。注意：用戶曾說「單斜不用扣15」，被他自己的實單推翻；後又說「−15 是之後要改的」——**目前維持扣 15，這條規則未來可能再改，改之前要拿新實單數字** |
| 斜樑支數 | 固定 `4`（雙C合併，左右各一組，2組×2支），與跨距/坡度無關 | 👁→✅ 用戶明確口述現場規則＋本案場 4 支吻合；但只被**一個**案場驗證過，大跨距是否要加中間支撐未知（見第 7 節） |
| 橫梁(桁條) | `purlinN = max(2, ceil(railCm/80)+1)`；間距 `railCm/(purlinN-1)`；總支數 `purlinN+1`（只加滴水側收邊 1 支，**不是**雙斜的 +2） | ✅ 465→7 支、77.5cm、共 8 支 |
| 中柱高 | `highSideH = round(fullWallCm * sr / 10)`，`fullWallCm = dir==='col' ? bd : bw`（**整個跨距，不含延伸**） | ✅ **但過程反覆橫跳三次，見下方警示框** |
| 中柱支數 | 固定 `4`（雙C合併左右各一組，同斜樑規則） | 👁→✅ 同斜樑支數的保留意見 |
| 浪板片數 | `pcsPerFace = ceil(pp/wW)`，`pp` 為橫梁長（垂直排水方向、含延伸） | ✅ ceil(660/75)=9 |
| 左右包角 | `flatCm + 10`，×2 支 | ✅ 480+10=490×2 |
| 橫向包角 | `perpCm + 20`；`>600cm` 對切成 2 支，否則 1 支 | ✅ 660+20=680>600→340×2。**但 600 門檻與 +10/+20 裕度是否為通用常數，未確認（第 7 節）** |

> ⚠️ **中柱高公式的「反覆橫跳」事件（memory 已標「之後不要再改」）：**
> 最初寫整個跨距（4m×1分水=40cm）→ 用戶給「40 是兩分水、20 是一分水」→ 反推成**半跨距**並改了程式＋memory → 用戶說「我說錯了」，重新給數字「**400cm 單斜，1 分水＝40cm、0.5 分水＝20cm**」→ 改回**整個跨距**，Fable5 複查定案。
> 教訓寫進了 `feedback_verify_formulas_with_numbers`：**公式只認具體輸入輸出數字反推，不認口頭描述；而且反推定案過的結論仍可能被用戶事後修正，每次質疑都要重新拿數字核對，不能說「上次對過了」。** 如果有人（包括用戶）再要求改這條公式，先要一組數字反推，再動手。

### 3.2 雙斜（rType===2）——大部分沿用舊版，實單驗證程度遠低於單斜

| 項目 | 公式（`calcMats()` rType===2 分支，約 :1082-1140） | 驗證狀態 |
|---|---|---|
| 斜面板材長 | `flatA = round(bd/2+ovF)`、`flatB = round(bd/2+ovB)`（row 方向則用 bw/2+ovL/ovR） | ❓ 幾何推導，無實單 |
| 斜樑長 | `flat - 15` | ❓ 單斜的 −15 理由是滴水，雙斜的理由是「中脊蓋住接合處」；是否應為 −30（中脊15＋滴水15）曾被提出，**未解決** |
| 斜樑支數 | `n = ridgeColN`（直接抄中脊柱組數，見下一列，**不再**呼叫 `bestCSpacing()`） | ✅ 2026-07-08 定案，見下方「斜樑支數＝中脊柱組數」警示框 |
| 斜樑間距（顯示用） | `ridgeSpanCm = dir==='col'?bw:bd`；`space = ridgeColManual ? null : ridgeSpanCm/(n-1)` | ✅ 2026-07-08 修正，見下方警示框——舊版誤用 `flatA/flatB`（斜面跑道長，維度整個錯） |
| 中柱（中脊柱）高 | `ridgeH = round(halfWallCm * sr / 10)`，`halfWallCm = dir==='col' ? bd/2 : bw/2`（半跨距、不含延伸） | ✅ 幾何上與單斜整跨公式自洽（例：600/2×2分水/10=60cm），且經 3D 呈現無異議 |
| **中脊柱組數** | `ridgeColN = 手動標記數 ‖ 固定 2`（約 :1131-1138） | 👁 **本工具最重要的一次「視覺驗證抓到假公式」案例，見下方警示框** |
| 中柱支數 | `ridgeColN × 2`（雙C合併每組 2 支） | 👁 用戶口述「雙斜也是一樣，鋼構常見併雙C柱」 |
| 橫梁總支數 | `totalC = 前後斜樑支數加總 + 2 支封邊` | ❓ 無實單 |
| 包角 | 每面 `flatCm + 30`，×2 支 | ❓ 無實單；單斜實單已證明包角有兩類且公式不同，雙斜這條單一公式很可疑 |
| 中脊配件 | 總長 <7m 出 1 支；否則對切各 `長/2+30`；琉璃瓦用半圓中脊＋筵口瓦＋梅花頭 | ❓ 沿用 v1 舊邏輯 |

> ⚠️ **中脊柱組數事件（7 組 vs 2 組）——「視覺表示本身就是驗證工具」的實證：**
> 這個值曾長期預設「跟斜樑支數一致」（`faces[0].n`，bestCSpacing 算出 400×600 是 **7 組**）。這個假設聽起來合理（跟斜樑對稱）、也一直沒人質疑，因為材料清單上只是一個數字。直到 3D 示意圖把 7 根紅色中柱**畫出來**，用戶一眼說「一組組合屋的中柱一般都兩組，你畫太多了」——才發現這是個從未驗證過、實際上就是錯的預設。修正為固定 2 組（左右各一組，與單斜慣例對齊），手動落柱標記優先權不變。
> **含義：對「✅實單驗證」的公式，改動門檻是新數字證據；對「❓未驗證」的公式（上表雙斜一整排），要抱持它隨時可能被下一張實單或下一張 3D 截圖推翻的心態，用戶質疑時優先相信用戶的現場經驗。**

> ⚠️ **斜樑支數＝中脊柱組數，且斜樑的「間距」是中柱的間距，不是斜樑自己算的（2026-07-08）：**
> 承上一條事件修正完中脊柱組數之後，斜樑支數（`faces[0].n`）當下**還是**獨立用 `bestCSpacing(flatA/flatB)` 算，只是這次剛好也常常等於 2，容易被誤以為已經一致。用戶馬上追問：「斜樑確實是依照中脊的隻數來決定，所以我才希望可以手動設定中柱」——結構上根本原因是**中柱撐起斜樑，斜樑才讓橫梁擺得上去**：中柱在哪、斜樑就在哪，兩者是同一組數字，不是碰巧算出來一樣。改法：斜樑支數 `n` 直接 = `ridgeColN`，`bestCSpacing()` 不再被雙斜呼叫（函式定義保留，留給未來「手動落柱時的建議間距」功能參考，目前無呼叫點）。
> 同一輪還抓到第二個問題：材料清單顯示斜樑「首隻+1=2隻（間距X cm）」，這句話的措辭（「首隻＋N-1」）其實是**橫梁(桁條)沿板材長度等分配置**的講法，不該套在斜樑上——斜樑不是沿自己的板材長找間距，是跟著中柱的位置走。而且舊公式 `space=(flat-30)/(n-1)` 用的 `flat`（斜面跑道長 bd/2 一類）維度也錯了：斜樑（中柱）是沿**中脊線方向**（垂直於跑道方向，dir='col' 時是 bw、dir='row' 時是 bd）分布的，這才是 3D 圖裡 `ridgeStart→ridgeEnd` 紅柱實際用的維度。修正為 `ridgeSpanCm=dir==='col'?bw:bd`，`space=ridgeSpanCm/(n-1)`（例：600×400 轉向、dir='col'、bw=600 → 2組間距600cm，即兩端各一組）；手動落柱標記時位置不一定等分，直接顯示「依手動落柱位置」不給假的等分數字。措辭也從「首隻＋N-1＝N隻」改成「與中柱同組數＝N隻」，跟橫梁的措辭區隔開。
> **含義：兩個數字「巧合相等」不代表邏輯已經綁在一起——如果背後算法是各自獨立的公式，遲早會在某個案例分岔。真正綁定要讓其中一個直接讀另一個的值，不要各自算完再指望結果一樣。**

### 3.3 公式的「第三份複本」：`draw()` 裡的即時數字標籤

2D 俯視圖每棟右上角箭頭旁的「跨距400→中柱40cm」「半跨300→中脊30cm」即時數字（`draw()` 內約 :489-492 單棟、:547-549 自動合併框），是**故意從 `calcMats()` 抄一份**的（註解寫明：避免每次重繪整包重算材料）。所以中柱高/中脊高公式實際上存在**三份**：`calcMats()`、`calc3DUnitDims()`、`draw()` 的標籤。改公式必須三處同步（見第 6 節）。

## 4. 三層混淆陷阱——本專案最重要的一課，加任何新視覺/標籤前必查

這三個坑不是理論風險，每一個都在本專案真實發生過、耗掉大量除錯時間。**任何時候你要在這個工具上加新的視覺元素、文字標籤、方向指示，先逐一對照這三條再出手，不要只「看一眼覺得對」就出貨。**

### 4.1 陷阱一：「同一個字母代表不同東西」（W/D 字母陷阱）——本專案踩了三次

一個可轉向（`state`）建物有兩套 W/D：**轉向前的身分 W/D**（`b.w/b.d`，永遠是規格上的 400×600）和**轉向後的畫面 W/D**（`bDims()` 的 `bw/bd`，轉向就互換）。兩套值印在畫面上都寫成「W」「D」，字面完全相同、指的東西完全不同。

三次踩坑實錄：
1. **全域下拉選單事件**：舊設計用一個全域「排水沿D邊」下拉選單，「D」指轉向後的有效 D；但畫布建物標籤印的是原始 W/D。用戶把 4×6m 建物轉向後選「沿D邊」，以為是沿 4m 邊算，結果中柱算出 60cm（用了 6m 邊）。Fable5 診斷出根因後整個廢掉全域選單，改成 per-building `dir` ＋畫布上直接點的箭頭＋即時數字。
2. **3D 邊長標籤事件（同一個坑自己重新挖一次）**：修完上面之後，3D 圖新功能加了「W600cm」「D400cm」邊長標示——但接的數字是 `uw/ud`（轉向後 bbox）。建物一轉向，「D」字母對應的 cm 數字整個換掉。用戶傳截圖圈出「底下標反了，難怪問題一直無解」才抓到。修法：**拿掉字母，只印純 cm 數字**，用顏色（橘＝目前排水方向那條邊）辨識方向。
3. 3D 屋頂正上方的「uw×ud」摘要文字同樣中招（轉向一次 400×600 變 600×400，讓人以為尺寸變了），改成單棟優先顯示原始 `w×d`。

**規則：在任何可轉向/可交換的物件附近，禁止使用裸的「W」「D」字母標籤。要嘛印純數字＋顏色/位置辨識，要嘛明確寫「原始」「畫面」。**

### 4.2 陷阱二：「翹高的邊」vs「跑道方向」互相垂直（單斜幾何直覺陷阱）

單斜屋頂上，肉眼看到「被墊高、沿自身維持等高」的那條稜線，跟「長度真正決定中柱高的那段水平跑道（低邊走到高邊的距離）」**剛好互相垂直**。人看 3D 圖會自然去讀稜線：「4 米那邊翹高了，所以是沿 4 米算」——但沿 4 米算的意思是跑道＝4 米，即稜線應該是 6 米那條。用戶就這樣連續多輪覺得「圖是對的、數字是錯的」，實際上兩者一致，是讀圖方向反了。

修法（現行實作，`draw3D()` rType===1 分支）：直接在屋頂斜面上畫一條**白色箭頭沿真正的爬升方向**（低邊中點→高邊中點），旁邊印「跑道600→中柱60cm」，語彙跟 2D 俯視圖的「跨距→中柱」同一套。教訓：**數字/箭頭要疊在使用者眼睛真正在看的東西上（屋頂斜面本身），放在底邊的小字標籤沒有人會去對照；同一個概念全工具只准用一套講法。**

### 4.3 陷阱三：斜投影的比例只在 90° 倍數視角下成立（數學結構限制，不是 bug）

`project3D()`（約 :1423）是 cavalier 斜投影：`sx = rx*k + ry*cos(20°)*k`，`sy = ry*sin(20°)*k − z`，k=0.5。這條公式經歷過三次修正，每次的結論都要記住：

1. **軸角色反轉**（subagent 第一版 bug）：把高度 z 畫成斜移、深度 y 畫成直上——垂直牆面在畫面上變斜的。正解：**深度 y 斜移（往畫面裡退）、高度 z 純垂直**，旋轉不影響 z，牆的垂直邊在任何視角都必須真的垂直。
2. **兩軸縮放不對稱**：原公式 x 軸不乘 k、y 軸合成長度乘 k，兩軸每 cm 的螢幕長度是 2:1——600cm 的 D 邊畫出來比 400cm 的 W 邊還短，直接騙過用戶的長度判斷，是整個排水方向除錯一直卡住的深層原因。正解：x 軸也乘同一個 k，兩軸 screen-sensitivity 相等，W:D 螢幕比例精確等於真實 cm 比例（驗證：400→200px、600→300px）。
3. **任意角度旋轉必然失真（已數學驗證，不要再試）**：先繞 Z 軸轉 (x,y) 再套上面的投影，只有旋轉角是 **90° 倍數**（0/90/180/270）時 W/D 比例才精確；45°、37° 等任意角度比例一定跑掉。這是斜投影（非等角投影）的線性變換結構限制——它只對公式原始設計的兩個特殊方向等比例；就算換成真 isometric 也只保證三個座標軸方向等比例，不保證任意對角線方向。**所以旋轉按鈕限制成只能 90° 跳（`rotate3D`），這是設計決策不是偷懶。任何人想加「自由旋轉視角」，只有兩條路：限制 90° 步進，或接受一般角度下比例會說謊（那就違背了「圖本身不騙人」的既定目標）——要嘛就整個換 three.js 真 3D（原規劃的第 3 步，目前刻意不做以維持零相依）。**
4. 附帶：傾角從 30° 降到 20°（`tilt=Math.PI/9`）是視覺舒適度調整——30° 時 D 邊的長度有一半耗在「往上爬」，肉眼用橫向佔比判斷長度會覺得 6m 沒有比 4m 長。15°/10° 試過太扁、失去立體感，20° 是定案值。

## 5. 驗證方法論（本專案實際使用、且必須延續的做法）

### 5.1 兩條鐵律

### 5.2 Playwright 驗證管線（已安裝，改完必跑）

工具位置：`C:\Users\lan77\.claude\tools\browser-check\`（Playwright + headless Chromium，獨立於本 repo）。

- 快速截圖：`node screenshot.js "<本機檔案路徑或URL>" "<輸出png路徑>" [寬] [高]`，路徑不用加 `file:///` 前綴，輸出 png 用 Read 工具看。
- 進階（互動測試）：自己寫臨時 node 腳本 require 該目錄的 playwright，`page.goto('file:///C:/Users/lan77/ping-calc/combo-roof-v2.html')`。

**測試時的狀態操作 vs 真實互動，界線要分清楚：**

- **佈局/設定 setup** 可以用 `page.evaluate()` 直接改 state（例：`buildings.push({x:0,y:0,w:400,d:600,type:'modular',state:2,groupId:null,dir:'col'}); draw();`、`document.getElementById('slopeR').value=1`），這只是搭場景，快又穩。
- **但凡測的是命中判斷或互動路徑本身**（點方向箭頭、點中脊線加落柱標記、拖曳、多選模式），**必須用真的 `page.mouse.click()` / `page.tap()` 打真實座標**，不准用 evaluate 直接改 state 冒充——手機「多選模式攔截中脊線點擊」「端點把手搶走落柱點擊」這兩個真實 bug，都只有真實事件路徑才重現得出來。

**從世界座標(cm)換算到 Playwright 螢幕點擊座標的完整公式**（`hitTest`/`hitRidgeLine` 的逆運算）：

```js
// 在 page.evaluate 裡取得換算所需的值：
const cv = document.getElementById('cv');
const rect = cv.getBoundingClientRect();
// 世界座標 (cx, cy)（cm）→ 瀏覽器 client 座標：
// canvas 內部 px = PAD + (cx - _ox) * _sc        （PAD=40，_sc/_ox/_oy 是 draw() 每次重建的全域）
// client X = rect.left + 內部px * (rect.width / cv.width)   ← 內部像素與CSS像素比例不一定是1！
// client Y = rect.top  + (PAD + (cy - _oy) * _sc) * (rect.height / cv.height)
```

三個必踩的坑：① `cv.width`（canvas 內部解析度）≠ `rect.width`（CSS 顯示尺寸），縮放比例一定要乘；② `PAD=40` 常數別漏；③ **點擊前先 `cv.scrollIntoView()`**，畫布常在視窗下方，不捲進來 click 會落在視窗外整個沒反應（實際踩過）。

**回歸基準場景**（改任何算料相關的東西，至少重跑第一個）：
1. 實單案例：400×600、state=2、dir='col'、單斜、前50/後30/左30/右30、1 分水 → 應得斜樑 465×4、橫梁 660×8（間距77.5）、浪板 480×9、左右包角 490×2、橫向包角 340×2、中柱 40cm×4。
2. 單棟 4×6m 單斜切換 dir：col=中柱60、row=中柱40（跟 3D 白色箭頭文字一致）。
3. 兩棟 400×600 貼齊自動合併 → 只出現 1 個方向箭頭，點擊兩棟 dir 同步變，算料用 800cm 合併跨距。
4. 十字型三棟（左300×600、中400×600轉90、右300×600）手動分組 → 合併成 1200×700、材料清單標「手動合併3間」。
5. 雙斜 400×600 → 中脊柱「自動計算 2 組／4 隻」，3D 紅柱數量＝清單組數。
